# LoveChoice / Buding 项目规则

这是一个语音 AI 对话系统项目，核心链路：

ASR → LLM → TTS

## 项目结构

```
buding/
├── web/
│   ├── web_server.py      # FastAPI 主服务
│   ├── runtime_brain.py    # 记忆管理、工具管理、对话逻辑
│   ├── static/
│   │   ├── index.html      # 对话页
│   │   ├── services.html   # 服务管理页
│   │   ├── settings.html   # 配置页
│   │   ├── app.js          # 前端主逻辑（三页共享）
│   │   └── styles.css      # 全局样式
│   └── runtime/            # 运行时数据（对话、日志、记忆数据库）
├── tts/
│   └── trained_tts_server.py
├── scripts/
│   └── start_voice_web.sh
└── logs/
```

## 开发原则

1. 优先保持项目简单清晰。
2. 不要随意破坏现有后端 API（/api/config, /api/services, /api/conversations, /api/memory, /api/tools, /api/health）。
3. 不要随意修改 WebSocket 协议（/ws/dialog）。
4. 不要随意修改配置字段名。
5. 前端优化要注意统一风格、响应式布局和可维护性。
6. 新增功能前先说明方案。
7. 大改前必须等待用户确认。
8. 修改后必须说明改了哪些文件、如何测试。
9. 复杂逻辑要有必要注释。
10. 不要引入复杂工程化工具，除非用户明确同意。

## 开发工作流（自动触发）

以下规则适用于任何需要读代码、改代码、增删文件的任务。纯聊天问答不受影响。

### 触发条件

当用户说以下内容时，自动进入 AI Change Flow 工作流：
- 任何涉及修改代码的请求（"帮我改"、"修复"、"优化"、"重构"、"加个功能"等）
- 任何涉及创建/删除文件的请求
- `/ai-change-flow`

### 工作流强制规则

**最顶层铁律（比其它所有规则优先级更高）：**

1. **一轮回复只能处理一个阶段。** 输出当前阶段结果后必须停止。不要在同一轮回复里提前执行下一阶段的工作（包括思考下一阶段）。
2. **每个阶段必须先输出可见的标题和内容（如"## 阶段 1：需求确认"），再等待用户回复。** 不能把多阶段合并到一轮回复里。
3. **每个阶段你都要输出一段带标题的结构化文字，然后明确问用户是否可以继续。** 用户说可以才能进入下一阶段。
4. **六个阶段一个不能跳过，任务再简单也必须走完。**
5. **阶段 0-2 只读只分析，阶段 3 才开始改代码。**
6. **阶段 4 和 5 不能省略，即使你认为"审查通过没问题"也必须输出审查报告和最终文档。**

**违反以上规则的具体表现（必须避免）：**

- ❌ 在一轮回复里同时输出阶段 1、2、3 的内容
- ❌ thinking 块里把"方案设计"做完然后直接跳到"执行修改"
- ❌ 修改完代码后不输出阶段 4 审查和阶段 5 文档
- ❌ 用户说"帮我改个东西"后，直接改完只汇报结果，跳过了 1/2/4/5

**六阶段流程：**

```
阶段 0: Skill Discovery — 检查需要的 skills，缺的搜索建议
阶段 1: 需求确认 — 复述需求、明确目标、确认范围
阶段 2: 方案设计 — 给出方案、文件级计划、风险点
阶段 3: 执行修改 — 按方案改代码
阶段 4: 代码审查 — 检查质量、防回归
阶段 5: 最终文档 — 修改说明、测试方法
```

**启动时第一句话必须输出（这算第一轮回复，只输出这个）：**

```
## AI Change Flow 启动

严格按以下流程，每阶段完成后等你确认：
0. Skill Discovery → 1. 需求确认 → 2. 方案设计 → 3. 执行修改 → 4. 代码审查 → 5. 最终文档

现在进入阶段 0。
```

**各阶段必须输出的内容模板**见 `.claude/skills/ai-change-flow/SKILL.md`。

### 什么情况下不触发

以下情况不需要走工作流，直接回答即可：
- 纯知识问答（"这个函数是干什么的"、"怎么用 Python 读文件"等）
- 聊天/闲聊
- 简单的文件查看请求（"帮我看一下这个文件"）
- 用户明确说"直接改不用流程"
