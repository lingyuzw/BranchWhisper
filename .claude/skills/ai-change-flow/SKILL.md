# AI Change Flow

你是主控 AI。这个 skill 的核心目标：**让用户掌控每一步节奏，AI 不能擅自跳过。**

## 最顶层铁律（比其它所有规则优先级更高）

### 规则 1：一轮回复只处理一个阶段

**每轮你只能做一件事：输出当前阶段的结构化文字，然后问用户是否可以继续。然后你**必须停止**，等用户下一轮回复。**

你**不能**做这些事：
- ❌ 在阶段 0 的输出里顺便想了阶段 1 的内容
- ❌ 在阶段 1 的输出里顺带给出了阶段 2 的方案
- ❌ 在 thinking 块里"因为前面阶段都确认了，我直接跳到执行"
- ❌ 把阶段 3 执行、阶段 4 审查、阶段 5 文档合并到一条回复里

你**只能**这样做：
- ✅ 输出当前阶段内容 → 明确问"确认后进入下一阶段？" → 停止 → 等用户回复

### 规则 2：用户必须用明确的"确认类"回复才能放行

**用户说以下任一短语才算确认，你才能进入下一阶段：**
- "可以"、"行"、"好"、"对"、"是的"、"没问题"
- "继续"、"下一步"、"进入下一阶段"、"下一阶段"
- "确认"、"同意"、"OK"、"yes"、"go ahead"

**用户说以下内容不算确认，你需要追问：**
- 只发数字（如"1"、"13"）— 这不明确，要追问"你是说确认进入阶段 1，还是想表达别的意思？"
- 只发"嗯"或"哦" — 可能是敷衍，要再问一遍
- 混入了新需求 — 要先确认当前阶段是否通过，还是需要重来
- 任何不含上述确认词的短回复

### 规则 3：每个阶段必须输出带标题的结构化内容

输出格式必须是 `## 阶段 X：XXX` 开头，包含该阶段规定的所有小节。

### 规则 4：禁止以"简化流程"为理由跳阶段

不管任务多简单、多明显，0→1→2→3→4→5 一个不能少。你觉得"审查肯定没问题"也得输出阶段 4 审查报告。

---

## 工作流总览

```
用户需求
→ 阶段 0: Skill Discovery — 检查需要的 skills
→ 用户明确确认（如"继续"）
→ 阶段 1: 需求确认 — 复述需求、分析影响、确认目标
→ 用户明确确认
→ 阶段 2: 方案设计 — 给出方案、文件级计划、风险点
→ 用户明确确认
→ 阶段 3: 执行修改 — 按方案改代码
→ 阶段 4: 代码审查 — 独立审查（3→4 自动过渡但必须输出）
→ 阶段 5: 最终文档 — 修改说明、测试方法（必须输出）
```

## 启动

触发 skill 后，**第一轮回复只输出以下内容，然后停止：**

```
## AI Change Flow 启动

严格按以下流程，每阶段完成后等你确认：
0. Skill Discovery → 1. 需求确认 → 2. 方案设计 → 3. 执行修改 → 4. 代码审查 → 5. 最终文档

现在进入阶段 0。
```

---

## 阶段 0：Skill Discovery

**本阶段只做 skill 检查和搜索建议，不分析需求，不改代码。**

输出模板（单独一轮回复，输出后等用户确认）：

```
## 阶段 0：Skill Discovery

### 当前任务类型
...

### 本地已有 skills
...

### 建议使用的 skills
...

### 缺失的 skills
（有就列，没有就说"无需额外 skills"）

### 搜索建议
（有就列，没有就说"不需要搜索"）

---
确认后进入阶段 1（需求确认）？或者回复"跳过"直接进阶段 1。
```

---

## 阶段 1：需求确认

**本阶段只分析需求、读代码理解现状，不改任何代码。**

输出模板（单独一轮回复，输出后等用户确认）：

```
## 阶段 1：需求确认

### 我理解的需求
...（3-5 句话）

### 模糊点
...（有就列，没有就说"无模糊点"）

### 必须实现
...

### 可选优化
...

### 可能影响范围
...

### 确认问题
...（有就问；没有说"无需额外确认"）

---
以上理解是否正确？确认后进入阶段 2（方案设计）。
```

---

## 阶段 2：方案设计

**本阶段只设计方案，不改代码。方案中要区分本次做和以后做。**

输出模板（单独一轮回复，输出后等用户确认）：

```
## 阶段 2：方案设计

### 总体思路
...

### 本次必须完成
...

### 建议顺手优化（本次做）
...

### 后续建议（以后做）
...

### 文件级修改计划
| 文件 | 修改内容 | 风险 |
|------|---------|------|

### 不应修改的内容
（API、DOM ID、配置字段、WebSocket 协议等）

### 测试计划
...

---
方案是否可行？确认后进入阶段 3（执行修改）。
```

---

## 阶段 3：执行修改

**按确认方案修改代码。本阶段可以连续执行工具调用，但全部改完后必须输出阶段报告。**

改完后必须输出：

```
## 阶段 3：执行修改（完成）

### 修改文件清单
| 文件 | 修改说明 |
|------|---------|

### 是否影响原有接口
...

### 快速验证方法
...

进入阶段 4（代码审查）
```

---

## 阶段 4：代码审查

**必须单独一轮输出。不能跳过，不能合并到阶段 3 或 5。**

```
## 阶段 4：代码审查

### 结论
（通过 / 需要返工）

### 需求完成度
...

### 是否破坏现有功能
...

### 代码质量问题
...

### 返工项
...

### 测试建议
...
```

---

## 阶段 5：最终文档

**必须单独一轮输出。这是整个流程的收尾。**

```
## 阶段 5：最终文档

# 修改完成说明

### 本次目标
...

### 修改文件清单
| 文件 | 修改说明 |
|------|---------|

### 新增功能
...

### 是否影响原有接口
...

### 启动和测试方法
...

### 已知风险
...

### 后续优化建议
...

### 建议提交信息
git commit -m "..."
```

## 触发场景

以下任何表述都触发本流程：
- "帮我优化前端"、"帮我加功能"、"帮我整理结构"、"帮我重构"、"帮我修 Bug"、"帮我改"、"修复"
- `/ai-change-flow`

## 速查表

| 阶段 | 做什么 | 改代码？ | 需确认？ | 单独一轮？ |
|------|--------|---------|---------|-----------|
| 0 | 检查 skills | ❌ | ✅ 需确认 | ✅ |
| 1 | 需求分析 | ❌ | ✅ 需确认 | ✅ |
| 2 | 方案设计 | ❌ | ✅ 需确认 | ✅ |
| 3 | 执行修改 | ✅ | ❌ 自动过渡 | ⚠️ 改完输出报告 |
| 4 | 代码审查 | ❌ | ❌ 自动过渡 | ✅ |
| 5 | 最终文档 | ❌ | ❌ | ✅ |

**阶段 3、4、5 虽然不需要用户确认，但**每个阶段必须单独输出一轮**，不能合并。**
